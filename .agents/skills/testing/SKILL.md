# Testing & Verification Guide

This skill provides guidelines and commands for running existing tests, writing new tests, and
compiling/executing Provar E2E tests.

---

## Provar E2E Test Structure

Provar utilizes a graph-based testing approach. An E2E test is a YAML graph that compiles to a
runnable TypeScript file the engine executes against a real browser via Playwright.

### Where tests live

Tests live inside a **Provar project** — a folder that contains a `.provar/` directory with a
`config.yml` and a `tests/` subfolder. Two ways to get a project:

1. **Scaffold one with the CLI** (recommended for new projects):
   ```bash
   bun --cwd apps/provar-cli -- init my-app --sample
   cd my-app
   ```
   This creates `my-app/.provar/tests/<name>.test.yml` ready to compile.

2. **Clone the bundled sample** (recommended for the first five-minute run):
   ```bash
   git clone https://github.com/thani-sh/demo-social.git ~/Code/provar-sample/demo-social
   cd ~/Code/provar-sample/demo-social
   ```
   The sample repo lives in the external [`thani-sh/demo-social`](https://github.com/thani-sh/demo-social)
   repo, not inside this monorepo — `docs/PRODUCT.md` § 4 calls out the split.

Either way, your working directory when you run `compile` / `run` should be the project root (the
folder that *contains* `.provar/`). Test paths below are written relative to that root.

### Test file layout

Each test consists of two parts, both living next to each other under `.provar/tests/`:

1.  **Graph Schema (`.test.yml`)** — test metadata, nodes, branching flows. Authored by hand
    or generated by the editor canvas.
2.  **Implementation (`.test.ts`)** — the compiled, runnable TypeScript. Generated by
    `provar compile`; do not hand-edit.

### Writing tasks in `.test.ts`

The compiled `.test.ts` file is generated by the engine; you don't write it by hand. The engine
emits this shape (see `libs/engine/src/compiler/compiler.ts` for the exact producer):

```typescript
import type { TestAPI } from "@libs/engine";

export const tasks = {
  ["task_login_step"]: async (api: TestAPI) => {
    await api.page.goto(api.var.baseUrl);
    await api.page.locator('input[placeholder="Username"]').fill(api.var.user.username);
    await api.page.locator('input[placeholder="Password"]').fill(api.var.user.password);
    await api.page.locator('button:has-text("Sign In")').click();
    await api.expect(api.page.locator('#post-composer-textarea')).toBeVisible();
  },
};
```

The `TestAPI` surface — defined in `libs/engine/src/types.ts` — exposes Playwright's `page`, an
`expect` assertion handle, a `var` map of project variables, and a `state` map for cross-task
data. Use Playwright locators and assertions via `api.page` and `api.expect` inside task functions.

## Provar CLI Commands

The canonical way to invoke the CLI is from the workspace root:

```bash
bun --cwd apps/provar-cli -- <command> [args...]
```

A shorthand also works via the root `package.json` script: `bun run provar <command>`. Both forms
call the same Bun entry point; the `--cwd` form is what the end-user docs
(`apps/provar-web/src/routes/docs/products/provar-cli`) use, so prefer it for documentation.

Run all commands from the project root (the folder that contains `.provar/`).

### 1. Scaffolding a new project

Creates a fresh `.provar/` project directory. `--sample` copies the bundled starter project
(`apps/provar-app/sample-projects/todo-app/`) so a new user can run a passing test with no further
configuration.

```bash
bun --cwd apps/provar-cli -- init <name|path> [--sample] [--force]
```

*Example:* `bun --cwd apps/provar-cli -- init my-app --sample`

### 2. Compiling tests

Compiles `.test.yml` files into runnable `.test.ts` files. Each compile writes a
`.trace.json` next to the output that captures per-step performance timing.

```bash
bun --cwd apps/provar-cli -- compile <test-file-path|dir> [--trace]
```

The path argument may be a single `.test.yml` file or a directory; in directory mode, the CLI
walks the project to discover every `.test.yml` and compiles them in order.

*Example:* `bun --cwd apps/provar-cli -- compile .provar/tests/auth/login-flow.test.yml --trace`

### 3. Running E2E test suites

Executes the compiled `.test.ts` files against a real browser via Playwright.

```bash
bun --cwd apps/provar-cli -- run <test-file-path|dir> [options]
```

*Options:*
- `--up-to <taskId>` — Run the E2E test execution only up to the specified task ID.
- `--headless <true|false>` — Set to `false` to run the browser in headed mode (default: `true`).

*Example:* `bun --cwd apps/provar-cli -- run .provar/tests/auth/login-flow.test.ts`

In directory mode the CLI discovers every compiled `.test.ts` under the given folder and runs
them in order.

### Common full-loop one-liner

```bash
bun --cwd apps/provar-cli -- compile . && bun --cwd apps/provar-cli -- run .
```

`compile` requires an LLM API key in the environment (e.g. `OPENAI_API_KEY`); `run` does not for
compile-only debugging, but does need Playwright's Chromium installed (`bunx playwright install
chromium`) for any task that touches a real browser.

## Unit & Integration Tests

For library functions (e.g., in `libs/engine`, `libs/domain`, `libs/config`, `libs/models`), use
Bun's native test runner. Test files live under `__tests__/` next to the source they cover and
match the source filename (e.g. `src/foo.ts` → `src/__tests__/foo.test.ts`).

- To run all tests in the workspace:
  ```bash
  bun run test
  ```
  (Runs `bun --filter '*' test` — every workspace package's `test` script.)

- To run tests in a specific package:
  ```bash
  bun --filter @libs/engine test
  ```
  or, equivalently, from inside the package directory:
  ```bash
  cd libs/engine && bun test src
  ```

- To run a single file or pattern:
  ```bash
  bun test libs/engine/src/__tests__/compiler.test.ts
  ```
