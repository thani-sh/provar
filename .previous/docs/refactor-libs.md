## 1. Executive Summary & Code Health Audit

### 1.1 Diagnosis (one paragraph)

The four libraries (`@libs/config`, `@libs/domain`, `@libs/engine`, `@libs/models`) are *mostly
disciplined* — their dependency graph is a clean DAG, there are no circular imports, and there is
exactly one curated public/private split (engine, per ADR 015). The codebase is **not bloated at the
library level** (3,431 LOC total, with the largest file at 492 LOC). The bloat lives in two specific
patterns: **(a) over-exposed barrels** that re-export symbols no consumer imports, and **(b) the
`@libs/engine/internal` barrel itself**, which has accreted 20 named re-exports since ADR 015 was
written. The biggest single risk is that internal helpers and dead classes are reachable from app
code (only prevented today by `tsc` discipline, not by package boundaries), so any rename silently
breaks contract without anyone noticing.

### 1.2 Health metrics

| Metric | `@libs/config` | `@libs/domain` | `@libs/engine` | `@libs/models` | Total |
|---|---|---|---|---|---|
| Source LOC | 454 | 278 | 2,393 | 306 | **3,431** |
| Test LOC | 382 | 77 | 874 | 99 | 1,432 |
| Files | 4 src + 2 test | 2 src + 1 test | 12 src + 5 test | 5 src + 2 test | 34 |
| Files > 300 LOC | 0 | 0 | 4 | 0 | 4 (all in engine) |
| Public exports (post-ADR-015) | 16 | 11 | 15 | 9 | **51** |
| Public exports actually consumed | 12 | 7 | 9 | 7 | **35** |
| Wasted export ratio | **25%** | **36%** | **40%** | **22%** | **~31%** |

> "Public exports actually consumed" is the count of distinct symbols imported by at least one file
> outside the library itself. The gap is the dead-export population.

### 1.3 Top 5 systemic risks

| # | Risk | Severity | Why it bites |
|---|---|---|---|
| 1 | `@libs/engine/internal` barrel is uncurated — has drifted to 20 re-exports since ADR 015 | **High** | Apps can accidentally depend on internals; one app already does (`editor-store.svelte.ts:8` imports `buildGraphPaths` from `@libs/engine`, not from `@libs/engine/internal`, contradicting the ADR). |
| 2 | Three copies of `MUTATING_METHODS` exist in `@libs/engine` (T017, T062) | **High** | Drift-prone; one is in a "dead" class that the test still covers. |
| 3 | Two `ProviderConfigError` classes with the same name in different libs (T077) | **Medium** | Forces every consumer to alias (`ProviderConfigError as ConfigProviderError`, `ProviderConfigError as ModelsProviderError`). |
| 4 | Domain zod barrel exposes 11 schemas; only 4 are imported by apps | **Medium** | Dead surface area invites future contributors to depend on schemas that may be reshaped for internal reasons. |
| 5 | Dead `MutationTrackingPage` class + its test = 328 LOC of unreachable code | **Medium** | The class claims a bug fix that never landed; the underlying monkey-patch is still in `test-run.ts` and `compiler/generator.ts`. |

### 1.4 What is NOT a problem

- No circular dependencies between libraries.
- No app reaches into `libs/<name>/src/internal/...` deep paths (no escape-hatch imports).
- No dead `@apps/*` dependencies in `package.json` (each dep is used; the only "dead dep" is `@libs/domain` in `apps/provar-cli`, which is a `package.json` hygiene issue, not a code issue — Phase 8.1).
- The `TestAPI` type is intentionally public (it is emitted into generated `.test.ts` files via
  `import type { TestAPI } from "@libs/engine"` at `compiler/compiler.ts:258`).

### 2.8 Surface-violation already in production

`apps/provar-app/src/mainview/lib/stores/editor-store.svelte.ts:8`:

```ts
import { buildGraphPaths } from "@libs/engine";
```

`buildGraphPaths` is **not** exported from `@libs/engine`. It is exported only from
`@libs/engine/internal`. Either:

(a) `tsc --noEmit` is failing silently and we should know about it; or
(b) the build pipeline (Vite/Svelte/Bun) is resolving through some lenient path; or

The fix is to change the import to `@libs/engine/internal`, **and** to verify with a clean `tsc

## 3. Target Architecture & Pattern Mapping

### 3.1 North-star principle: "If an app imports it, the public surface documents it; otherwise it's private."

Today:

```text
                 ┌─────────────────────────────────────────────┐
                 │           App (provar-app, cli)             │
                 └────────────────┬────────────────────────────┘
                                  │
        ┌─────────────────┬───────┴───────┬──────────────────┐
        ▼                 ▼               ▼                  ▼
  @libs/config     @libs/domain      @libs/engine       @libs/models
        │                │               │ │ (./internal)
        │                │               │ │
        └────────────────┴───────────────┘ │
                                          ▼
                                   (no app imports
                                    from @libs/engine/internal)
```

Target:

```text
                 ┌─────────────────────────────────────────────┐
                 │           App (provar-app, cli)             │
                 └────────────────┬────────────────────────────┘
                                  │
        ┌─────────────────┬───────┴───────┬──────────────────┐
        ▼                 ▼               ▼                  ▼
  @libs/config     @libs/domain      @libs/engine       @libs/models
  (curated)        (curated)         (curated)          (curated)
        │                │               │
        └────────────────┴────── @libs/engine/internal ────┐
                                (curated, internal-only)    │
                                                          ▼
                                                   engine-internal
                                                   modules only
```

The **boundary between public and internal** becomes a *physical* contract, not a *review-time*
contract. Apps that try to import an internal symbol from `@libs/engine` will get a `tsc` error
because the symbol isn't in the public barrel.

### 3.2 Public-surface rules (formalized from ADR 015)

These rules apply uniformly to all four libraries, not just `@libs/engine`:

1. **Every export in `src/index.ts` must be reachable from an app file** (CLI or app).
   - Run `pnpm dlx ts-prune` or equivalent in CI to enforce. (Phase 8.2)
2. **No `export *` anywhere in any library's public barrel.**
   - Curated named exports only.
3. **One "deep" subpath per library, maximum.**
   - `@libs/engine/internal` exists; no other library should grow a `internal` subpath unless
     needed.
4. **Test-only exports live behind `__test__` namespaces** (existing convention in `@libs/config`
   and `@libs/models`).
5. **If a symbol is renamed or removed in a library, a `grep -r` across `apps/` confirms no broken
   import** before the commit lands.

### 3.3 Pattern: barrel-as-curated-API

Today `@libs/engine`'s `src/index.ts` follows this pattern. The other three libraries don't have
the discipline — `@libs/config` has 16 public symbols, 4 of which are dead; `@libs/domain` has 11
public symbols, 7 of which are dead (counting schemas only used internally). The Phase 2 work
applies the same barrel discipline to all four libraries.

### 3.4 Pattern: Strategy for the duplicate `ProviderConfigError` (T077)

The two errors represent distinct failure modes (configuration incomplete vs. no API key), so they
should remain separate classes. But the **names must differ**. Pattern:

```ts
// libs/config/src/schema.ts
export class ConfigProviderError extends Error {
  readonly provider: ProviderName;
  readonly requirements: ProviderRequirement[];
  // ...
}

// libs/models/src/registry.ts
export class ModelProviderError extends Error {
  readonly provider: AgentClientConfig["provider"];
  // ...
}
```

CLI catch sites become:

```ts
try { … } catch (e) {
  if (e instanceof ConfigProviderError) { … }
  if (e instanceof ModelProviderError) { … }
}
```

No aliases. No ambiguity. (Phase 6.4)

### 3.6 Pattern: types-only barrel for `@libs/domain`

`@libs/domain` has two exports subpaths (`.` and `./zod`). The cleaner shape — and the target —
is **one types subpath** (`./zod` keeps its name for backward compatibility but everything in it
moves to `./types`), and the `"."` entry is dropped (T032). All app imports become
`@libs/domain/zod` or `import type { … } from "@libs/domain/zod"`.

> Renaming `./zod` to `./types` is left out of scope for now (it touches every app import). If a
> future contributor wants to do it, see `docs/refactor-other-apps.md`.

### 3.7 Pattern: dead-export lint (Phase 8.2)

A CI step in `bun run lint:unused-exports` (or equivalent) runs [`ts-prune`](https://github.com/nadeesha/ts-prune)
across each lib's `src/index.ts` and fails the build on any export that no other lib or app imports.
This makes "wasted export ratio" go to 0% and stay there.

// libs/engine/src/runtime/tracker.ts (new file)
import type { Page } from "playwright";

export const MUTATING_METHODS = [
  "click", "fill", "type", "press", "goto",
  "check", "uncheck", "selectOption", "hover", "dblclick",
] as const;

export type MutatingMethod = (typeof MUTATING_METHODS)[number];

export interface MutationTracker {
  readonly hasMutated: boolean;
  reset(): void;
}

export function trackMutations(page: Page): MutationTracker {
  let hasMutated = false;
  for (const method of MUTATING_METHODS) {
    const original = page[method].bind(page);
    page[method] = ((...args: unknown[]) => {
      hasMutated = true;
      return (original as (...a: unknown[]) => unknown)(...args);
    }) as typeof page[typeof method];
  }
  return {
    get hasMutated() {
      return hasMutated;
    },
    reset() {
      hasMutated = false;
    },
  };
}

## Appendix C — Sequencing rationale

- Phase 1 must come first (it's a real bug).

  recommended order is 2.1 → 2.2 → 2.3 → 2.4 → 2.5 → 2.6 → 2.7 → 2.8 because earlier removals
  reduce the noise for later ones.
